# Arch Engine Schema Notes (V1.1)

## Purpose
Schema enforces architecture-decision behavior and auditable lifecycle transitions so runtime cannot degrade into generic CRUD.

## Constraint Rationale

### `features`
- `name UNIQUE`: canonical feature identity.
- `status` constrained to `active|deprecated|experimental`.
- partial unique index on active `capability_key`: single active lineage head.
- unique `(capability_key, version)`: monotonic lineage guard surface.

### `feature_candidates`
- staging layer for extraction output before activation.
- unique `(ingestion_record_id, capability_key)`: deterministic dedup per ingestion.
- `candidate_status` + `resolver_decision`: explicit resolver lifecycle trace.
- reusability gate fields:
  - `reusable_score` (0..10)
  - `reusable_across_projects` (0/1)
  - `reusable_reason`
- dependency trace fields:
  - `inferred_dependencies`
  - `unresolved_dependencies`

### `feature_scores`
- one score row per feature (`feature_id PRIMARY KEY`).
- `score_total` generated by DB for deterministic resolver/query math.

### `feature_relations`
- relation vocabulary: `replaces|replaced_by|variant|depends_on`.
- duplicate relations prevented with unique triple.
- self-loop blocked.

### `feature_history`
- append-only snapshots for replacement audit.
- update/delete blocked by triggers.

### `ingestion_records`
- classification constrained to `runtime-linked|non-operational|standalone-reference`.
- cleanup telemetry stores decision/mode/result/error/timestamps.
- state check enforces valid pending/applied transitions.

## Resolver + Planner Readiness
- candidate indexes support ingestion status scans and reusability filtering.
- `features(category,status)` + `feature_scores(score_total DESC)` support best-active queries.
- dependency edges in `feature_relations` provide composability graph (`depends_on`).

## Write-Path Enforcement Split
- DB guarantees structural integrity.
- runtime enforces policy sequence:
  - reusability gate before active promotion,
  - replacement snapshot + monotonic version increment,
  - dependency linking and unresolved dependency capture,
  - keep-old/discarded traceability.

## Cleanup Safety
- default deletion only for `non-operational` + `cleanup_decision=delete`.
- runtime-linked cannot be auto-deleted in default mode.
- all cleanup actions must persist execution outcome for audit.
